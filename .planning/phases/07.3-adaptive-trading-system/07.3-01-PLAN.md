Build a contract classifier that parses Kalshi ticker strings to extract resolution dates and classifies contracts into monitoring tiers based on time-to-resolution. Handles 8+ distinct ticker formats found across the 615 active pairs.

Purpose: The adaptive trading system needs to know when each contract resolves to determine bar collection frequency (15min to daily) and position management rules. 374 of 615 tickers lack embedded month/day, requiring either sports/politics defaults or Kalshi API fallback.

Output: `src/live/contract_classifier.py` with TDD test suite, plus `data/live/pair_classifications.json` after running on all 615 pairs.

<!-- Ticker format analysis from active_matches.json (615 pairs, 95 unique prefixes): -->

Format 1 - YYMMMDD with threshold suffix (daily oil/BTC):
  KXWTI-26APR08-T105.99       -> resolves April 8, 2026
  KXWTIW-26APR10-T118.99      -> resolves April 10, 2026
  KXBTCD-26MAR3122-T67099.99  -> resolves March 31, 2026 (note: "3122" = day31 + extra chars)

Format 2 - YYMMMDD with alphanumeric suffix (monthly):
  KXBTCMAXMON-BTC-26APR30-7500000   -> resolves April 30, 2026
  KXBTCMINMON-BTC-26APR30-6500000   -> resolves April 30, 2026

Format 3 - YYMMMDD yearly:
  KXBTCMAXY-26DEC31-149999.99  -> resolves December 31, 2026
  KXBTCMINY-27JAN01-55000.00   -> resolves January 1, 2027

Format 4 - YYMM only (Fed meetings):
  KXFEDDECISION-26APR-C25     -> resolves end of April 2026
  KXFED-27APR-T0.00           -> resolves end of April 2027
  KXFEDCOMBO-26APR-0-0        -> resolves end of April 2026

Format 5 - Year-only elections (354 tickers!):
  KXPRESNOMD-28-AOC           -> Kalshi API says close_time: 2028-11-07T15:00:00Z
  KXPRESNOMR-28-MR            -> convention date ~Aug 2028
  KXVPRESNOMD-28-AOC          -> convention date ~Aug 2028

Format 6 - Year-only sports:
  KXNBA-26-SAS                -> NBA finals ~June 2026
  KXMLB-26-PHI                -> World Series ~Oct 2026
  KXMENWORLDCUP-26-US          -> WC final July 19, 2026
  KXF1-26-CL                  -> season end ~Dec 2026
  KXUCL-26-ATM                -> UCL final ~June 2026
  KXLALIGA-26-BAR             -> season end ~May 2026
  KXEUROVISION-26-AUS         -> ~May 2026

Format 7 - Split YY-MMMDD:
  KXTRUMPCHINA-26-MAY15       -> resolves May 15, 2026
  KXTRUMPCHINA-26-JUN01       -> resolves June 1, 2026

Format 8 - Special/no date parseable:
  KXFEDCHAIRCONFIRM-JSHE      -> no date, use Kalshi API /markets/{ticker}
  KXPERSONPRESMAM-45           -> no date parseable
  KX2028DRUN-28-CBOOK         -> year only, special

Distribution: 156 with full YYMMMDD, 65 YYMM-only, 374 year-only, ~20 truly unparseable

<!-- Tier classification thresholds (from planning guidance, updated from CONTEXT.md): -->
MONITORING:  > 7 days (> 168 hours)     -> bar_interval = daily (86400s)
MONTHLY:     30-90 days                  -> bar_interval = 4h (14400s)
WEEKLY:      7-30 days                   -> bar_interval = 1h (3600s)
DAILY:       0-7 days                    -> bar_interval = 15min (900s)

NOTE: These are the user-specified tiers from the plan_structure_guidance.
The CONTEXT.md has finer tiers (ACTIVE_DAY, SPRINT, etc.) but the guidance
simplifies to 4 tiers: daily, weekly, monthly, quarterly.
Dynamic reclassification: re-evaluate tier each cycle based on current time.

<!-- Kalshi API for resolution dates of unparseable tickers: -->
GET https://api.elections.kalshi.com/trade-api/v2/markets/{ticker}
Response: {"market": {"close_time": "2028-11-07T15:00:00Z", ...}}
Rate limit: 20 req/s. Need this for ~374 year-only + ~20 unparseable tickers.
Cache results to data/live/resolution_dates_cache.json to avoid repeated API calls.

    - parse_resolution_date(ticker: str) -> datetime | None:
      1. Try full YYMMMDD: regex r'(\d{2})(JAN|FEB|MAR|APR|MAY|JUN|JUL|AUG|SEP|OCT|NOV|DEC)(\d{2})' anywhere in ticker
         - Handle BTCD special case: if match has extra digits after day, just take first 2
      2. Try split YY-MMMDD: regex r'-(\d{2})-(JAN|FEB|MAR|APR|MAY|JUN|JUL|AUG|SEP|OCT|NOV|DEC)(\d{2})$' at end
      3. Try YYMM only: regex r'(\d{2})(JAN|FEB|MAR|APR|MAY|JUN|JUL|AUG|SEP|OCT|NOV|DEC)(?:-|$)' -> last day of month
      4. Try year-only: regex r'-(\d{2})-' -> check SPORTS_DEFAULTS and POLITICS_DEFAULTS by prefix
      5. Return None if nothing matches

    - fetch_resolution_from_api(ticker: str) -> datetime | None:
      GET /trade-api/v2/markets/{ticker}, parse close_time field
      Cache results in self._api_cache dict (persisted to data/live/resolution_dates_cache.json)
      Rate limit: sleep(0.06) between calls
      Return None on 404 or error

    - classify_contract(resolution_date: datetime | None, now: datetime) -> Tier:
      If resolution_date is None: return Tier.UNKNOWN
      days_remaining = (resolution_date - now).total_seconds() / 86400
      if days_remaining <= 7: Tier.DAILY
      elif days_remaining <= 30: Tier.WEEKLY
      elif days_remaining <= 90: Tier.MONTHLY
      else: Tier.QUARTERLY

    - classify_all_pairs(matches: list[dict], now: datetime | None = None, use_api: bool = True) -> dict[str, dict]:
      For each match:
        1. parse_resolution_date(kalshi_ticker)
        2. If None and use_api: fetch_resolution_from_api(kalshi_ticker)
        3. classify_contract(resolution_date, now)
        4. Store: {pair_id: {"tier": tier.name, "resolution_date": iso_str|None, "days_remaining": float, "bar_interval_seconds": int, "kalshi_ticker": str}}
      Save API cache after all lookups

    - CLI (__name__ == "__main__"):
      Load active_matches.json, classify_all_pairs, print distribution table, save to pair_classifications.json
      --no-api flag to skip API lookups (faster for testing)
