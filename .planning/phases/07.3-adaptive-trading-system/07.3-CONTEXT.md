# Phase 7.3: Adaptive Trading System

**Purpose:** Replace the dumb "bet and close every bar" system with a real trading engine that adapts bar frequency to contract lifecycle, manages multi-bar positions, and uses intelligent entry/exit rules.

<domain>
## Phase Boundary

**In scope:**
- Dynamic bar interval based on time-to-resolution (4h → 1h → 15min → 5min ramp)
- Contract classifier (parse resolution dates, assign to time horizon tier)
- Position manager (persistent state across cron runs, tracks open positions)
- Entry rules (contract ≥ $0.10, spread ≥ 30pp, model signal)
- Exit rules (take profit, stop loss, time stop, momentum exit, resolution proximity, trailing stop)
- Oracle Cloud deployment (15min cron with adaptive logic)
- PPO-based exit policy (RL learns when to exit instead of hardcoded rules)

**Out of scope:**
- Real money execution (still paper trading)
- New prediction models (reuse existing LR, XGBoost, GRU, LSTM)
- Kalshi/Polymarket execution APIs (future phase)
</domain>

<decisions>
## Implementation Decisions

### Dynamic Bar Intervals (Adaptive Frequency)

Contracts ramp up in monitoring frequency as they approach resolution. This captures the acceleration of spread convergence near expiry.

| Time to Resolution | Bar Interval | Bars/Day | Rationale |
|-------------------|-------------|----------|-----------|
| > 7 days | 4h | 6 | Slow-moving, just monitoring |
| 3-7 days | 1h | 24 | Warming up, position entry zone |
| 1-3 days | 15min | 96 | Active trading, convergence accelerating |
| 6h - 24h | 5min | 288 | Sprint to resolution, fast moves |
| < 6 hours | 5min + EXIT flag | 288 | Close positions before resolution risk |
| < 2 hours | NO NEW ENTRIES | — | Too close, resolution outcome uncertain |

**Collection strategy:** Oracle Cloud cron runs every 5 minutes. Each run:
1. Load all active pairs with resolution dates
2. Classify each pair into a tier based on time remaining
3. Only collect bars for pairs whose tier interval has elapsed since last collection
4. e.g., a 4h-tier pair only gets a new bar every 4 hours, even though the cron runs every 5min

This way ONE cron job handles all tiers efficiently.

### Contract Classification

Parse resolution date from Kalshi ticker and Polymarket data:
- Kalshi: `KXWTI-26APR08-T105.99` → resolves April 8, 2026
- Kalshi: `KXBTCMAXMON-BTC-26APR30-7500000` → resolves April 30, 2026
- Polymarket: `end_date` field in market metadata

Classification:
```python
def classify_contract(resolution_date, now):
    hours_remaining = (resolution_date - now).total_seconds() / 3600
    if hours_remaining < 2:
        return "NO_TRADE"      # too close to resolution
    elif hours_remaining < 6:
        return "SPRINT"        # 5min bars, exit flag
    elif hours_remaining < 24:
        return "ACTIVE_DAY"    # 5min bars
    elif hours_remaining < 72:  # 3 days
        return "ACTIVE_WEEK"   # 15min bars
    elif hours_remaining < 168: # 7 days
        return "WARMING"       # 1h bars
    else:
        return "MONITORING"    # 4h bars
```

### Position Manager (Persistent State)

Tracks open positions across cron runs via JSON state file.

**State file:** `data/live/positions.json`
```json
{
  "live_0042": {
    "pair_id": "live_0042",
    "kalshi_ticker": "KXWTI-26APR08-T105.99",
    "direction": "short_spread",
    "entry_spread": 0.52,
    "entry_time": "2026-04-08T14:30:00Z",
    "entry_kalshi_price": 0.80,
    "entry_poly_price": 0.28,
    "current_spread": 0.41,
    "bars_held": 12,
    "max_spread_since_entry": 0.55,
    "min_spread_since_entry": 0.38,
    "unrealized_pnl": 0.11,
    "stop_loss": 0.78,
    "take_profit": 0.15,
    "trailing_stop": 0.50,
    "tier": "ACTIVE_WEEK",
    "resolution_date": "2026-04-10T23:59:00Z"
  }
}
```

**Each cron run:**
1. Load positions.json
2. Fetch latest prices for all open positions
3. Update current_spread, unrealized_pnl, bars_held
4. Check exit conditions for each position
5. If exit triggered → close position, log to paper_trades.jsonl, remove from positions
6. Check for new entry opportunities (pairs not already in a position)
7. If entry triggered → add to positions.json
8. Save positions.json

### Exit Rules (Rule-Based Starting Values)

| Rule | Condition | Default Value | Rationale |
|------|-----------|---------------|-----------|
| **Take Profit** | spread converged below threshold | spread < 10pp | Lock in gains when disagreement largely resolved |
| **Stop Loss** | spread widened beyond limit | spread > entry × 1.5 | Cut losses if platforms diverge further |
| **Trailing Stop** | spread rebounds from best level | spread > best + 15pp | Lock in partial gains on pullback |
| **Time Stop** | held too long | 24h for daily, 72h for weekly | Don't hold through resolution |
| **Momentum Exit** | 4 consecutive bars moving against | 4 bars against direction | Trend reversal signal |
| **Resolution Exit** | contract resolving soon | < 2 hours to resolution | Avoid binary resolution risk |
| **Profit Target %** | captured enough of the spread | captured > 60% of entry spread | Alternative to absolute take-profit |

**For short_spread positions (betting spread narrows):**
- Entry: spread = +0.50 (Kalshi 0.80, Poly 0.28)
- Take profit: spread < +0.10 → close, P&L = +0.40
- Stop loss: spread > +0.75 → close, P&L = -0.25
- Trailing stop: spread hit +0.20, now back to +0.35 → close, P&L = +0.15

### PPO Exit Policy (Phase 2 of this system)

After rule-based exits are running, train PPO to learn optimal exits:

**State:** (current_spread, entry_spread, spread_change_last_bar, bars_held, hours_to_resolution, unrealized_pnl, max_spread, min_spread)

**Actions:** {HOLD, EXIT}

**Reward:**
- EXIT with profit → +realized_pnl
- EXIT with loss → realized_pnl (negative)
- HOLD when spread converges next bar → small positive (delayed gratification)
- HOLD when spread diverges next bar → small negative

**Training:** Use historical spread trajectories from live data (once we have 5+ days). Each episode = one position from entry to exit/resolution. PPO learns timing patterns humans miss.

### Oracle Cloud Deployment

**VM:** Oracle Linux 9, 1 OCPU, 1GB RAM, 129.80.4.112

**Architecture:**
```
Oracle VM (cron every 5 minutes):
  └── python run_trading_cycle.py
        ├── Load positions.json
        ├── Classify all pairs by time-to-resolution
        ├── Collect prices for pairs whose bar interval has elapsed
        ├── Update open positions with new prices
        ├── Check exit rules → close triggered positions
        ├── Check entry rules → open new positions
        ├── Log trades to paper_trades.jsonl
        ├── Save positions.json
        └── Every hour: git commit + push results

GitHub Actions (keep running hourly as backup):
  └── Same collection but hourly only
  └── Fallback if Oracle VM goes down
```

**Memory constraint (1GB):**
- Don't load PyTorch/GRU/LSTM on the VM — too heavy
- Use only Linear Regression + XGBoost for signals (scikit-learn + xgboost = ~200MB)
- Pre-train models locally, save as pickle, deploy to VM
- This is fine — LR is the best model after fees anyway

**Model save/load (needed for VM deployment):**
- Add `save(path)` and `load(path)` to BasePredictor
- Save trained LR + XGBoost as pickle files
- VM loads pickled models instead of retraining every cycle
