<domain>
## Phase Boundary

Replace broken candlestick APIs with trade-first data pipeline. Pull raw trades from both platforms, reconstruct 4-hour OHLCV+VWAP candles with microstructure features, and align cross-platform prices with forward-fill and 24-hour staleness decay. Output: `data/processed/aligned_pairs.parquet` with 50-70 usable pairs.

</domain>

<decisions>
## Implementation Decisions

### Granularity
- **4-hour candles** as primary granularity (not 1-minute, not hourly)
- With ~0.1 trades/hour on Kalshi, 4-hour gives ~30% fill rate vs 9% at hourly
- Only emit candles for bars that contain trades during reconstruction
- Forward-fill happens in the alignment step, not reconstruction

### Primary Price
- **VWAP** (volume-weighted average price) as primary price for spread computation
- OHLC retained as supplementary features
- `VWAP = sum(price_i * volume_i) / sum(volume_i)`

### Cross-Platform Alignment
- Forward-fill each platform independently on unified 4-hour grid
- **24-hour staleness limit** (6 bars) — after 24h with no trade, price = NaN
- Include `hours_since_last_trade` as feature for models
- **Do NOT interpolate** — forward-fill only (interpolation leaks future info)

### Trade Data Sources
- **Kalshi**: `/markets/trades` endpoint, cursor pagination, 200/request
- **Polymarket**: Data API `/trades` endpoint, offset pagination, 500/request, max 15000 offset
- Ditch candlestick APIs entirely for both platforms

### Microstructure Features Per Candle
- vwap, open, high, low, close, volume, trade_count, dollar_volume
- buy_volume, sell_volume (from taker_side field)
- realized_spread (mean ask hits - mean bid hits)
- max_trade_size, time_since_last_trade, has_trade

### Data Quality
- Price bounds: [0.01, 0.99]
- Minimum 20 trades per market (else exclude)
- Staleness ratio < 80% per pair (else exclude)
- Spread mean-reverting check: mean |spread| < 0.3
- Settlement convergence check near resolution dates

### Storage
- Raw trades: `data/raw/{platform}/{market_id}_trades.parquet`
- Aligned output: `data/processed/aligned_pairs.parquet` (single file, snappy compression)
- Quality report: `data/processed/data_quality_report.json`

### Input
- 77 accepted pairs from `data/processed/accepted_pairs.json`
- Market metadata from `data/raw/kalshi/_metadata.json` and `data/raw/polymarket/_metadata.json`

### Claude's Discretion
- Internal module structure within the 4 components
- Exact error handling and retry logic (reuse existing ResilientClient)
- Logging verbosity and progress reporting
- Parallelization strategy for trade fetching

</decisions>

<specifics>
## Specific Ideas

- Kalshi trades endpoint returns: created_time, yes_price_dollars, count_fp, taker_side, ticker
- Polymarket Data API trades return: timestamp, price, size, side
- Both have pagination (Kalshi: cursor, Polymarket: offset)
- Kalshi tested: KXU3-26FEB-T4.3 has 942 trades, reconstructs to 278 hourly candles
- Polymarket already has 77 parquet files from CLOB — can supplement with trade data or keep CLOB where it works
- Forward-fill alignment tested concept: 24h window should take usable pairs from 1 to 50-70

</specifics>

## Deferred Ideas
