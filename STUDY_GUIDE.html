Cross-Platform Prediction Market Arbitrage

DS340 Final Project — Complexity vs. Performance in ML-Driven Trading

"Does increasing model complexity (RL, anomaly detection) improve arbitrage detection over simpler regression approaches, and if so, when is that complexity justified?"

Answer: No. Simpler models (Linear Regression, XGBoost) beat complex models (GRU, LSTM, PPO) across every metric. Model complexity is not justified at this dataset scale.

How It Works: End-to-End Pipeline

The system detects price discrepancies between Kalshi and Polymarket on matched contracts, predicts whether spreads will converge or diverge, and generates trading signals.

1

Contract Matching (NLP)

Same events on both platforms are matched using semantic similarity. Keyword pre-filtering narrows candidates, then sentence-transformer embeddings (all-MiniLM-L6-v2, 384-dim) compute cosine similarity scores.

Kalshi: "Will WTI oil close above $70 on Apr 11?" Polymarket: "WTI Crude Oil above $70 on April 11?" Cosine similarity: 0.94 → MATCH
2

Spread Computation

Both platforms price the same binary event differently. The spread captures the price disagreement, which is our trading signal.

spread = kalshi_close - polymarket_close Example: Kalshi prices at $0.62, Polymarket at $0.55 Spread = +$0.07 (7 percentage points) → Kalshi thinks it's MORE likely than Polymarket
3

Feature Engineering (31 + 6 derived)

31 features from aligned OHLCV data on both platforms, plus 6 derived features: price velocity, volume ratio, spread momentum, spread volatility, and order flow imbalance for each platform.

Raw: OHLCV x 2 platforms, volume, trade count, dollar volume, buy/sell volume, realized spread, max trade size, has_trade, hours_since_last_trade, spread, pair_id Derived: price_velocity, volume_ratio, spread_momentum, spread_volatility, kalshi_order_flow_imbalance, polymarket_order_flow_imbalance
4

Spread Change Prediction

Models predict spread_change = next_spread - current_spread. Positive prediction means spread is expected to widen; negative means expected to narrow.

Target: y = spread[t+1] - spread[t] = (kalshi[t+1] - poly[t+1]) - (kalshi[t] - poly[t]) Regression models: predict exact value RL models: choose action {hold, long, short}
5

Trading Signal Generation

If |prediction| > threshold, enter a trade. Long the spread (buy Kalshi, sell Polymarket) if prediction is positive. Short the spread if negative. $100 per position.

if |prediction| > 0.02: # 2pp threshold if prediction > 0: LONG # expect spread to widen if prediction < 0: SHORT # expect spread to narrow else: HOLD # insufficient signal
6

Evaluation

Regression metrics (RMSE, MAE, directional accuracy) plus trading metrics (P&L, Sharpe ratio, win rate) with walk-forward backtesting and bootstrap confidence intervals.

Regression: RMSE, MAE, directional_accuracy Trading: total_pnl, num_trades, win_rate, sharpe_ratio Backtest: $10k capital, $100/pos, 5pp round-trip costs Bootstrap: 1000 resamples, 95% confidence intervals

Tier 1

Regression Baselines

Flat tabular features, no sequential modeling. Each bar is an independent observation. These are the strongest models in the project.

Linear Regression

Standard OLS regression predicting spread_change from 31 features. No regularization. Fits instantly.

Input(31 features) → LinearLayer(31, 1) → spread_change
No hyperparameters Closed-form solution scikit-learn

Best model after fees in live trading: +$5.83 net @2pp over 27 hours

XGBoost

Gradient-boosted decision trees. Strong tabular baseline that captures non-linear feature interactions automatically.

Input(31 features) → XGBRegressor(n_estimators, max_depth, ...) → spread_change
n_estimators=100 max_depth=6 learning_rate=0.1 subsample=0.8

Best RMSE overall: 0.286 | Best per-trade Sharpe: 0.52

Tier 2

Time Series Models

Recurrent neural networks that process sequences of 6 bars (24 hours at 4h resolution). Use warm-up stitching to give test rows full lookback context by concatenating training data tail.

GRU (Gated Recurrent Unit)

Single-layer GRU with input dropout and linear output head. Simpler than LSTM, fewer parameters, comparable performance.

Input(lookback=6, features=31) → Dropout(0.3) → GRU(hidden=64, layers=1) → Linear(64, 1) → spread_change
hidden_size=64 lookback=6 dropout=0.3 AdamW lr=1e-3 grad_clip=1.0 patience=10

LSTM (Long Short-Term Memory)

Single-layer LSTM with input dropout. Has separate cell state and hidden state (more parameters than GRU). Slightly better RMSE but more selective in trading.

Input(lookback=6, features=31) → Dropout(0.3) → LSTM(hidden=32, layers=1) → Linear(32, 1) → spread_change
hidden_size=32 lookback=6 dropout=0.3 AdamW lr=1e-3 patience=10

Warm-Up Stitching

Sequence models need lookback context. For the first few test rows, there's no preceding test data. We concatenate the tail of training data before the test data for each pair, so every test row gets a full 6-bar window.

# For each pair_id: train_rows = cached_train[pair_id] # tail of training set stitched = concat(train_rows, test_rows) for each test_row: window = stitched[pos-lookback : pos] # always 6 bars prediction = model(window)
Tier 3

Reinforcement Learning (PPO-Raw)

PPO agent acts directly on features without preprocessing. Tests whether RL itself adds value over regression—per Professor KG's feedback: "Could RL act directly on features?"

PPO-Raw (Proximal Policy Optimization)

Uses Stable-Baselines3 PPO with a custom Gymnasium environment. The agent observes a window of features and chooses hold/long/short. Reward includes 2pp transaction cost penalty.

SpreadTradingEnv(Gymnasium): State: flattened (198,) = 6 bars x 33 channels (31 features + current_position + time_fraction) Action: Discrete(3) = {hold, long, short} Reward: clip(position * spread_change - 0.02*|pos_change|, -0.5, 0.5) PPO Agent (Stable-Baselines3): Policy: MlpPolicy (default 2x64 hidden layers) Training: 100k timesteps across all pairs
SB3 PPO MlpPolicy lookback=6 tx_cost=0.02 reward_clip=0.5 100k steps

RMSE: 0.319 — worse than all Tier 1 and Tier 2 models. RL adds no value at this dataset scale.

Tier 4

RL + Anomaly Detection (PPO-Filtered)

Adds an autoencoder signal filter before PPO. The autoencoder flags anomalous spread patterns, and the RL agent only trades on flagged bars. Tests whether anomaly pre-filtering helps RL.

Autoencoder (Signal Filter)

Symmetric autoencoder trained on normal spread features. Bars with reconstruction error above the 95th percentile of training errors are flagged as anomalous.

Encoder: Linear(31→16, ReLU, BN) → Linear(16→8, ReLU, BN) → Linear(8→4) <-- bottleneck Decoder: Linear(4→8, ReLU, BN) → Linear(8→16, ReLU, BN) → Linear(16→31) Loss: MSE(input, reconstruction) Threshold: 95th percentile of training errors
bottleneck=4 Adam lr=1e-3 batch=32 patience=20 threshold=95th pctl

PPO-Filtered

Same PPO agent as Tier 3, but only receives bars that the autoencoder flagged as anomalous. Hypothesis: anomalous spreads represent better trading opportunities.

All bars → Autoencoder → flag anomalous (top 5%) Anomalous bars → PPO Agent → {hold, long, short} Normal bars → HOLD (no action)

Worst model: RMSE 0.327, per-trade Sharpe 0.01, lost money in bootstrap CI. The autoencoder filters out too many good trades and the RL agent can't learn from the reduced data.

Baselines

Naive Baselines (Lower Bound)

Naive: Spread Always Closes

Predicts spread_change = -current_spread. Assumes prices always converge to zero spread by resolution. RMSE: 0.500.

Volume: Higher Volume Correct

Predicts the higher-volume platform has the correct price. Direction of convergence follows volume. RMSE: 0.457.

⚠
Concerning finding: In live trading, the Naive baseline nearly matches XGBoost on gross P&L (+$14.32 vs +$16.59). This suggests the edge is primarily in the spread itself (prices DO converge), not in sophisticated prediction. The ML models add marginal value through better selectivity and timing.

Spread Threshold Ablation

Tested 0pp, 2pp, 5pp, and 10pp minimum prediction magnitude to trigger a trade. XGBoost dominates at every threshold. Higher thresholds increase per-trade quality but decrease total P&L due to fewer trades.

⚠
PPO limitation: PPO models produce discrete actions (hold/long/short), not continuous predictions. Threshold-based filtering doesn't work well—there's no prediction magnitude to threshold on. This is a fundamental design mismatch.

Key Insights

The Penny Contract Filter

The single most impactful engineering decision: filtering out contracts priced below $0.10 flips the strategy from losing to profitable at realistic fee levels.

Before Filter (All Contracts)

2,919 trades | Gross P&L: +$12.61
Net @2pp: +$3.17
Net @3pp: -$1.55 (LOSING)
Win Rate: 46.9% | W/L Ratio: 2.03x

After Filter (>= $0.10)

1,271 trades | Gross P&L: +$17.09
Net @2pp: +$8.67 (+174%)
Net @3pp: +$4.46 (PROFITABLE)
Win Rate: 48.8% | W/L Ratio: 2.92x

Why: Penny contracts ($0.01-$0.09) are near-certain outcomes (probability ~3% or ~97%). Spreads on these are noise, not signal. Fees eat 37% of gross on cheap contracts vs 10% on $0.50 contracts. 56% of all trades were on penny contracts.

Naive Baseline Implication

The Naive baseline (spread always closes) nearly matches XGBoost on gross P&L. This means the edge is primarily in the spread itself—prices DO converge—not in sophisticated prediction. ML models add marginal value through better selectivity and timing of entry.

Statistical Significance

All models (except PPO-Filtered) show statistically significant edge. 95% confidence intervals on mean P&L per trade exclude zero. But XGBoost/GRU/LSTM are statistically indistinguishable from each other—overlapping CIs on both RMSE and Sharpe.

Live vs Historical

In historical backtesting, XGBoost is the best model. In live paper trading, Linear Regression overtakes XGBoost. This is consistent with the thesis: simpler models generalize better to unseen data. XGBoost may be slightly overfitting to historical patterns.

Win Rate vs W/L Ratio

Win rates are only 51-52%, barely better than a coin flip. The strategy is profitable because winners are 1.6-2.3x larger than losers. This is a classic statistical arbitrage profile: low win rate, high payoff ratio, edge from positive expected value per trade.

Technical Details

BasePredictor Contract (All Models Implement This)

Every model inherits from BasePredictor and plugs into the shared evaluation pipeline. This ensures apples-to-apples comparison across all 8 models.

class BasePredictor(ABC): """Abstract base class for all spread-change predictors.""" @property @abstractmethod def name(self) -> str: ... @abstractmethod def predict(self, X: pd.DataFrame) -> np.ndarray: ... def fit(self, X_train, y_train) -> "BasePredictor": return self # no-op for baselines def evaluate(self, X_test, y_test, threshold=0.02, timestamps=None): predictions = self.predict(X_test) metrics = compute_regression_metrics(y_test, predictions) profit = simulate_profit(predictions, y_test, threshold, timestamps) return {**metrics, **profit}
Profit Simulation (simulate_profit)

Threshold-based directional trading simulator used by all models. Computes per-bar returns, cumulative P&L, and annualized Sharpe ratio.

# Strategy per timestep i: if |predictions[i]| > threshold: position = sign(predictions[i]) # +1 long or -1 short profit = actuals[i] * position # realized P&L else: profit = 0 # no trade (bar counted as 0 return) # Sharpe: annualized from daily returns # Panel-aware: aggregate by timestamp before computing # daily_returns = sum of bar_returns per calendar day sharpe = mean(daily_returns) / std(daily_returns) * sqrt(365)
Walk-Forward Backtester (Realistic Sharpe)

Steps through test bars chronologically with dollar-denominated positions and percentage-of-capital returns. Produces honest, paper-ready Sharpe ratios.

# Configuration: initial_capital = $10,000 position_size = $100 per trade entry_cost = 3pp of contract price exit_cost = 2pp of contract price # 5pp round-trip # For each bar: if |prediction| > threshold: num_contracts = $100 / mid_price gross_pnl = num_contracts * actual_change * direction net_pnl = gross_pnl - entry_cost - exit_cost # Daily return = sum(daily_pnl) / portfolio_value # Sharpe = mean(daily_pct) / std(daily_pct, ddof=1) * sqrt(365)
SpreadTradingEnv (Gymnasium for PPO)

Custom Gymnasium environment where the RL agent trades per-pair spread sequences.

SpreadTradingEnv(gymnasium.Env): # Observation: flattened window observation_space = Box(shape=(198,)) # 6 bars x 33 channels # 33 = 31 features + current_position + time_fraction action_space = Discrete(3) # 0 = hold (position 0) # 1 = long (position +1) # 2 = short (position -1) # Reward per step: raw_reward = position * spread_change - 0.02 * |position_change| reward = clip(raw_reward, -0.5, +0.5) # Episodes: each pair is one episode # Reset cycles through pairs in order

Quick Reference: What To Tell The TA

The One-Paragraph Summary

We built a cross-platform prediction market arbitrage system that matches contracts between Kalshi and Polymarket using NLP semantic similarity, computes price spreads, and predicts spread convergence using 8 models across 4 complexity tiers. We tested Linear Regression, XGBoost, GRU, LSTM, PPO-Raw, and PPO-Filtered (with autoencoder) on 144 matched pairs (6,802 training rows). The central finding is that model complexity does not improve performance: XGBoost (Tier 1) achieves the best RMSE (0.286) and Sharpe ratio (8.30), while PPO-Filtered (Tier 4) is the worst (RMSE 0.327, Sharpe -0.88 in bootstrap). We validated this with live paper trading on 598 pairs, where Linear Regression was the only model profitable after 3pp fees. The trading edge comes from oil/commodity contracts (15% of trades, 112% of profit) and is amplified by filtering out penny contracts (<$0.10).

What Worked

  • Simple models (LR, XGBoost) consistently outperform complex ones
  • Penny contract filter: single biggest improvement to P&L
  • Semantic matching pipeline: 615 live pairs from two disjoint APIs
  • Walk-forward backtester with honest Sharpe calculation
  • Bootstrap CIs proving statistical significance
  • Live paper trading validating historical results
  • Oil/commodity specialization: 66% win rate

What Did Not Work (And Why)

  • PPO-Filtered: autoencoder filters too aggressively, RL can't learn
  • PPO-Raw: RL adds no value over regression at this dataset scale
  • Long lookback windows: dataset too small for long-range dependencies
  • LSTM selectivity: fewer trades but worse per-trade quality
  • Penny contracts: noise drowns signal, fees eat profits
  • TFT (Temporal Fusion Transformer): deferred, too complex for data size
